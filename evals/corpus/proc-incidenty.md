# Управление инцидентами

## Классификация

| Severity | Критерий | Примеры |
|----------|----------|---------|
| SEV1 | Недоступен клиентский сервис или утечка данных | Прод лежит, массовые ошибки оплат |
| SEV2 | Деградация клиентского сервиса, есть workaround | Замедление API в 3+ раза |
| SEV3 | Внутренний сервис недоступен/деградирует | Не работает тестовый стенд |
| SEV4 | Единичная проблема без влияния на сервис | Ошибка у одного клиента |

## Роли

При SEV1/SEV2 назначается **инцидент-коммандер** (дежурный SRE-лид) —
он координирует работу и коммуникации; технические работы ведут дежурные
команды сервисов. Коммуникации со стейкхолдерами каждые 30 минут для SEV1
и каждый час для SEV2 — в канале #incidents.

## Процесс

1. Обнаружение (алерт или сообщение) → регистрация в «Астра-Инцидент»
   ботом `/incident new`.
2. Оценка severity инцидент-коммандером; при сомнении severity завышается.
3. Работы по восстановлению; фиксация таймлайна в треде инцидента.
4. Закрытие после подтверждения восстановления метриками.

## Постмортем

Для SEV1 и SEV2 постмортем обязателен в течение **5 рабочих дней**:
blameless-формат, таймлайн, корневые причины, план действий с владельцами
и сроками. Постмортемы публичны внутри компании и разбираются на
еженедельной встрече SRE.
