# Git-процесс разработки

## Модель веток

Используем trunk-based development: ветка `main` всегда деплоится,
фичи — в короткоживущих ветках вида `feat/ASTRA-1234-short-name`
(номер задачи из трекера обязателен).

```bash
git checkout main
git pull origin main
git checkout -b feat/ASTRA-1234-user-quota
# ... работа ...
git push -u origin feat/ASTRA-1234-user-quota
```

## Коммиты

Формат — Conventional Commits с номером задачи в теле:

```text
feat(billing): добавить квоты на пользователя

Задача: ASTRA-1234
```

Squash-merge в `main` — единственный разрешённый способ вливания:
история `main` состоит из одного коммита на задачу.

## Pull Request

- PR не более ~400 строк содержательного диффа; больше — дробить.
- Минимум **2 аппрува**, один из них — от владельца кода (CODEOWNERS).
- CI обязан быть зелёным; ветка живёт не дольше 3 рабочих дней.
- Самостоятельный merge автора после аппрувов; влитые ветки удаляются
  автоматически.

## Хотфиксы

Хотфикс идёт той же схемой из `main` с префиксом `fix/`, но допускает
1 аппрув дежурного ревьюера и внеочередной деплой (см. регламент деплоя).
