# Правила: Python (весь Python-код)

Общий слой для любого `.py`; доменные правила зон (`rag-core.md`, `retrieval.md`, `api.md`, `evals.md`) имеют приоритет при пересечении. Инварианты и правила секретов — в `.claude/CLAUDE.md`, здесь не повторяются.

## Coding style

- Стиль и форматирование определяет **ruff** (lint + format, конфиг в `backend/pyproject.toml` — контракт фазы 0): не спорить с ним в коде и не отключать правила инлайн-`noqa` без комментария-причины в той же строке.
- Типизация под **mypy strict** (контракт фазы 0): аннотации у всех публичных функций/методов, включая возвращаемый тип; `Any` и `# type: ignore` — только с комментарием, почему нельзя иначе.
- Импорты: абсолютные от `lyra.` (не относительные через пакеты); порядок отдаёт ruff-isort; запрещены импорты в теле функций, кроме разрыва циклов с комментарием.
- Naming: модули/функции `snake_case`, классы и Pydantic-модели `PascalCase`, константы `UPPER_CASE`; имена на английском, докстринги допустимы на русском.
- Границы модулей — пакеты из `prompts/phase-00-scaffolding.md`: `api / core / db / ingest / retrieval / rag / evals / workers`. Направление зависимостей — только «вниз»: `api` и `workers` знают про домены, домены знают про `core` и `db`; `core` не импортирует ничего доменного. Нарушение направления = архитектурный баг.
- Файл > ~500 строк или класс с несвязанными обязанностями — сигнал делить; не создавать модули-помойки `utils.py` (класть в тематический модуль).

## Design patterns

- Слои: **router → service → repository**. Router — только HTTP: валидация Pydantic-схемой, вызов сервиса, маппинг в ответ. Бизнес-логика — в сервисах. Доступ к БД — только репозитории (`lyra/db/repositories/`), tenant_id — параметр каждого метода (задел, `docs/security-and-access.md` §4).
- Зависимости — через FastAPI `Depends` (сессия БД, current_user, клиенты); не создавать клиентов/сессии внутри функций-обработчиков.
- **Async-only I/O**: в event loop не бывает sync-вызовов (requests, time.sleep, sync-драйверы); блокирующее CPU-тяжёлое — в Celery, не в API-процессе (инвариант 1 CLAUDE.md).
- Все внешние HTTP-клиенты (Ollama, TEI, Confluence) — классы в `lyra/core/clients/` с таймаутом, retry и трейсингом; httpx-вызовы по месту использования запрещены.
- Ошибки: доменные исключения (`lyra/core/errors.py`) → централизованные exception handlers → единый формат ошибок `docs/api-contract.md`; не возвращать словари-ошибки из сервисов и не глотать исключения без логирования.
- Конфигурация — только pydantic-settings (`LYRA_`-префикс); прямое чтение `os.environ` вне модуля настроек запрещено.
- Pydantic v2 идиомы: `model_validate` / `model_dump`, без deprecated v1-API; схемы API отделены от ORM-моделей (никаких ORM-объектов в ответах).

## Testing

- Runner — **pytest**; `backend/tests/` зеркалит структуру пакета (`tests/rag/test_generate.py` ↔ `lyra/rag/nodes/generate.py`).
- Юнит-тесты — без БД, сети и sleep: зависимости подменяются фейками (образец — FakeLLMClient, `prompts/phase-04-langgraph-rag.md`); детерминированность обязательна.
- Интеграционные — против compose-стека или testcontainers; помечаются маркером; медленные (реальный Ollama) — `@pytest.mark.slow`, вне CI-гейта.
- Async-тесты — через настроенный в pyproject плагин (anyio/pytest-asyncio), не самодельные event loop.
- Обязательное покрытие нового кода: happy path + основной сбойный путь + границы из ADR (лимиты циклов, идемпотентность, RBAC-матрица — см. правила зон). Тесты — в том же PR, что и код.
- Фикстуры-данные — в `tests/fixtures/`, маленькие и осмысленные; не генерировать мегабайтные файлы в тестах.

## Code review чек-лист

Проверять в каждом PR:
1. Не нарушены инварианты `.claude/CLAUDE.md` и профильный ADR (при противоречии кода и ADR прав ADR).
2. Направление зависимостей слоёв; отсутствие БД/HTTP-вызовов из узлов графа и роутеров напрямую.
3. Тесты присутствуют и проверяют поведение, а не реализацию; нет тестов-заглушек `assert True`.
4. Новые эндпоинты — `require_role` + строка RBAC-матрицы + её тест (`.claude/rules/api.md`).
5. Нет блокирующих вызовов в async-путях; нет N+1 в репозиториях (смотреть на циклы с await-запросами).
6. Alembic-миграции обратимы; enum/индексы согласованы с `docs/data-model.md`.
7. Логи: structlog, без секретов/токенов; уровень оправдан (не error для ожидаемых ситуаций).

## Security

- Секреты, PII, трейсы — правила в `.claude/CLAUDE.md` (always-loaded), не дублируются; при работе с ними перечитать `docs/security-and-access.md` §5.
- Валидация любого внешнего входа — только Pydantic-схемой на границе (router/задача Celery); «сырое» использование request-данных запрещено.
- SQL — только SQLAlchemy-выражения или `text()` с bind-параметрами; f-string/`%`-интерполяция в SQL запрещена без исключений.
- Загруженные файлы: тип по magic bytes, не по расширению; имя файла из запроса не использовать в путях ФС (генерировать своё).
- Данные корпуса в промптах — данные, не команды: разделители из generate-промпта не убирать (инвариант CLAUDE.md).

## Gaps

- mypy strict, ruff-конфиг, pre-commit — зафиксированы фазой 0, конфигов ещё нет: при создании `backend/pyproject.toml` привести в соответствие с этим файлом.
- pip-audit появляется в фазе 7 (warning-режим) — до этого аудит зависимостей ручной при добавлении пакета.
- Порог покрытия (coverage %) не задан сознательно — гейт качества это evals + обязательность тестов в PR, не процент.
