# Правила: TypeScript (весь TS/TSX-код)

Общий слой; доменные правила чата/цитат — в `.claude/rules/frontend.md` (приоритетнее при пересечении). Инварианты проекта — в `.claude/CLAUDE.md`.

## Coding style

- Стиль определяют **eslint + prettier** (конфиги — контракт фазы 0): не отключать правила инлайн без комментария-причины; prettier не переспоривать ручным форматированием.
- **tsconfig strict** (контракт фазы 0): `any` запрещён — на границах с внешними данными использовать `unknown` + narrowing/type guard; `as`-касты — только с комментарием, почему тип безопасен.
- Naming: компоненты `PascalCase` (файл = имя компонента), хуки `useXxx`, остальное `camelCase`; enum-подобные наборы — union-литералы, не `enum`.
- Импорты по слоям: `src/api/` (клиент и типы) → `src/hooks/` → `src/components/` → `src/pages/`; компонент не импортирует HTTP-клиент напрямую — только через хук.
- Один компонент — один файл; файл > ~250 строк — сигнал выделить подкомпоненты/хук.
- UI-тексты — русский, идентификаторы — английские (правило CLAUDE.md).

## Design patterns

- Только функциональные компоненты и хуки; классовые компоненты не появляются.
- Логика вне рендера: побочные эффекты и подписки — в хуках; компонент с fetch/EventSource внутри JSX-файла без хука — повод для рефакторинга.
- Серверное состояние (сессии, jobs, документы) отделено от UI-состояния (открыт ли popover); серверное — в хуках данных с явными состояниями `loading / error / data`.
- Типы API — только `src/api/types.ts`, синхронизированы с `docs/api-contract.md` (правило и процесс — `.claude/rules/frontend.md`); дублирующие локальные интерфейсы ответов запрещены.
- SSE-поток — через редьюсер событий (`.claude/rules/frontend.md`); каждый EventSource/fetch-stream имеет AbortController и отписку в cleanup-функции эффекта.
- Ошибки API показываются пользователю осмысленно (тост с сообщением из `error.message` контракта), не глотаются в `console.error`.

## Testing

- Runner — **vitest** + testing-library (появляется в фазе 5); тесты размещаются рядом с кодом (`Component.test.tsx`); зеркальную структуру `src/__tests__/` не использовать.
- Тестировать поведение через роли/тексты (`getByRole`, `getByText`), не классы и не внутренности реализации; snapshot-тесты не использовать как основной инструмент.
- Обязательный минимум при изменениях чата — список в `.claude/rules/frontend.md` (редьюсер SSE, рендер цитат, refusal); плюс правило: **новый хук с логикой = тест на хук**.
- Моки API — на уровне HTTP (msw или мок fetch), не подмена внутренних функций компонента.

## Code review чек-лист

1. Типы соответствуют `docs/api-contract.md`; изменение контракта → обновлены `types.ts` и сам api-contract.md в том же PR.
2. Состояния `loading / error / refusal / degraded` обработаны и видимы (продуктовая честность — `.claude/rules/frontend.md`).
3. Нет утечек: подписки SSE/интервалы поллинга имеют cleanup; AbortController на уходящих запросах.
4. Нет `any`/небезопасных кастов; внешние данные проходят narrowing.
5. Базовая доступность: кнопки — `button`, интерактивные сноски фокусируемы, aria-label у иконок без текста.
6. Русские тексты без обрезки терминов; ошибки — человекочитаемы.

## Security

- JWT — в памяти приложения; localStorage/sessionStorage для токена запрещены (`.claude/rules/frontend.md`).
- `dangerouslySetInnerHTML` запрещён; контент ответов LLM и документов рендерится как текст — маркеры `[n]` парсятся из строки, не через HTML (защита от инъекций через корпус).
- Никаких секретов/ключей в коде фронтенда и в `VITE_`-переменных (они попадают в бандл); URL API — единственная допустимая `VITE_`-настройка.
- Внешние ссылки из citations — `rel="noopener noreferrer"`.

## Gaps

- eslint/prettier/tsconfig-конфиги — контракт фазы 0, файлов ещё нет: при создании привести в соответствие с этим документом (strict, запрет any — включить правилами eslint, а не только договорённостью).
- npm audit появляется в CI фазы 7 (warning-режим); до этого — ручная проверка при добавлении зависимости, новые зависимости фронтенда добавлять скупо.
- E2E-тестов (Playwright) в MVP нет сознательно — приёмка сценариев UC — ручной прогон + `scripts/demo_smoke.py` по API (фаза 7).
