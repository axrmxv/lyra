# Правила: Git (глобально)

## Commits

- **Conventional commits**: `<type>(<scope>): <суть>`. Типы: `feat`, `fix`, `docs`, `test`, `refactor`, `chore`, `ci`, `perf`. Scope — зона проекта: `rag`, `retrieval`, `ingest`, `api`, `ui`, `evals`, `db`, `infra`; scope опционален для сквозных изменений.
- Заголовок — английский тип/scope, суть можно по-русски (принято первым коммитом репозитория); ≤ 50 символов, без точки в конце. Тело — по-русски: что и зачем (не «как» — это видно в diff), пустая строка после заголовка, строки ≤ 72 символов.
- Коммит атомарен: одна логическая единица; «and» в заголовке — сигнал разделить. Каждый коммит проходит `make test lint` (не коммитить заведомо красное, кроме WIP в личной ветке с последующим squash).
- Коммиты, сгенерированные с участием Claude, завершаются трейлером `Co-Authored-By: Claude <модель> <noreply@anthropic.com>`.
- Изменение промптов/retrieval/chunking в коммите → перед PR обязателен `make eval` (правило CLAUDE.md); упомянуть результат в теле коммита или PR.

## Commit suggestions (работа с Claude)

- `git add` / `git commit` / `git push` выполняет пользователь вручную; Claude их НЕ выполняет, а после каждого логического завершения работы (фича, фикс, рефакторинг модуля — не каждое сохранение файла) выводит готовые команды. Внутри большой задачи — на естественных checkpoint'ах.
- Формат вывода: bash-блок без комментариев внутри, готовый к копированию; в `git add` — конкретные файлы коммита, не `git add .`.
- Несвязанные изменения — отдельные атомарные коммиты в одном блоке, не один общий.
- Сообщения — по разделу «Commits» выше: `type(scope): subject`, императив, ≤50 символов, без точки; body через второй `-m` (что и зачем), если изменение неочевидно из subject.

## Branching

Модель: GitHub Flow. `main` всегда стабилен и деплоим. Любое изменение делается в отдельной короткоживущей ветке и вливается через Pull Request. НЕ коммить напрямую в `main`.

### Цикл работы

Одна задача (фича, фикс, рефакторинг модуля) = одна ветка = один PR.

```bash
# Ответвляйся всегда от свежего main
git checkout main
git pull origin main
git checkout -b feat/rag-retriever

# Работа и атомарные коммиты (см. commit-раздел выше)
git add backend/lyra/retrieval/retriever.py
git commit -m "feat(retrieval): add hybrid retriever"

# Публикация ветки
git push -u origin feat/rag-retriever

# После открытия PR, self-review и merge на GitHub:
git checkout main
git pull origin main
git branch -d feat/rag-retriever
```

До публикации репозитория на GitHub (remote не настроен) шаги `pull`/`push` пропускаются — остальной цикл действует.

Правила:
- Ответвляйся только от актуального `main` (`git pull` перед созданием ветки).
- Держи ветку короткой: в пределах одной задачи, merge в течение дня-двух. Чем дольше ветка живёт, тем больнее merge-конфликты.
- Дроби работу на маленькие вертикальные срезы. Гигантский PR невозможно нормально отревьюить.
- Удаляй ветку сразу после merge (локально `git branch -d`, на GitHub кнопкой при merge).

### Именование веток

Формат: `type/short-kebab-case`. Типы синхронизированы с Conventional Commits (см. commit-раздел выше).

| Префикс | Пример |
|---------|--------|
| `feat/` | `feat/rag-retriever` |
| `fix/` | `fix/empty-query-crash` |
| `refactor/` | `refactor/split-services` |
| `chore/` | `chore/bump-deps` |
| `docs/` | `docs/api-readme` |

Также допустимы `test/`, `ci/`, `perf/` — полный список типов в разделе «Commits».

Type ветки — основной type работы; вспомогательные коммиты других типов (`test`, `docs`, `refactor`) внутри ветки допустимы.

### Защита main

`main` подлежит защите branch protection после публикации репозитория на GitHub (см. Gaps): прямой push запрещён, изменения только через PR, перед merge проходят проверки CI и hooks. До настройки protection запрет прямых коммитов действует как дисциплина.

## PR / Review

- Размер: целиться в ≤ ~400 строк содержательного diff; генерённое (миграции, lock-файлы, фикстуры) — помечать в описании. Фазы из `prompts/` крупнее — дробить на PR по пунктам «Что реализовать», где это возможно.
- Описание PR: что сделано, зачем (ссылка на фазу PLAN.md / ADR / issue), как проверено (команды, скриншоты для UI); чек-листы ревью — `.claude/rules/python.md` и `.claude/rules/typescript.md`.
- Merge только при зелёном CI; **eval-гейт не обходить** и пороги не подгонять (`.claude/rules/evals.md`); красный nightly-eval — блокер следующего PR в затронутую зону.
- Self-review перед запросом ревью: пройти diff глазами, убрать отладочный код, случайные файлы и закомментированные блоки.

## Hooks

- pre-commit (ruff, prettier, gitleaks, commitlint — контракт `prompts/phase-00`) обязателен у всех: `pre-commit install` — часть онбординга в README.
- `git commit --no-verify` запрещён; исключение — только несрабатывающий hook по внешней причине, с объяснением в описании PR.
- Находка gitleaks = стоп: секрет не «переименовывается», а удаляется, ключ считается скомпрометированным и ротируется; если секрет попал в историю — переписать историю до пуша, запушенный — ротировать немедленно.
- В индекс никогда не попадают: `.env` и любые файлы с секретами, локальные данные (`volumes/`, дампы БД, модели), артефакты сборки. Изменения конфигурации — через `.env.example` с плейсхолдерами (правило CLAUDE.md).

## Gaps

- commitlint и gitleaks запланированы в фазе 0 (pre-commit hooks + CI job) — до её выполнения формат коммитов и отсутствие секретов держатся на ревью.
- Защита main (branch protection, required checks) настраивается в GitHub при появлении CI (фаза 0) — до этого правило действует как дисциплина.
- pre-commit-конфига ещё нет (контракт фазы 0): до его создания `make lint` вручную перед коммитом.
