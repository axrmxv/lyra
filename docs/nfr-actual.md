# Фактические NFR-замеры (фаза 7)

Стенд: ноутбук 16 ГБ RAM, WSL2 (12 ГБ), CPU-инференс (Ollama qwen2.5-7b q4,
TEI bge-m3 / bge-reranker-v2-m3). Дата прогона: 2026-07-17.
Методика: `scripts/load_smoke.py` (флаг `--unique` — обход кэша повторов,
замер холодного пути); цели — [nfr.md](nfr.md) §1–2.

**Честно про 20 RPS**: цель nfr §2 «10–20 RPS /search» на CPU-стенде не
достигается — узкое место embedding запроса (TEI CPU обрабатывает
последовательно, устойчивый предел ~3–4 RPS, дальше растёт очередь).
Пути (production-трек): GPU-инференс TEI, реплики embeddings-сервиса,
кэш эмбеддингов частых запросов. Функциональной деградации нет —
запросы выполняются, растёт латентность.

## Параллельные чаты (семафор LLM = 2)

3 диалога одновременно (`load_smoke.py chat --parallel 3`):

- диалоги 1 и 2 выполнились параллельно: **252 с и 298 с** до final
  (одиночный ~120–180 с — деградация ~×1.7 при делении CPU);
- диалог 3 мгновенно получил **429 + Retry-After** (семафор
  `llm_max_concurrency=2`) — защита локальной LLM работает по дизайну,
  очередь таймаутов не копится.

## Чистый старт (критерий ≤ 15 минут)

`docker compose down` + удаление data-volumes (pgdata, uploads,
hf_tokenizer) → `make up` → готовность всех зависимостей → `make seed-demo`
(22 дока корпуса + 4 витринных):

- READY (все health up): **189 с**
- seed-demo: **~31 с**
- **Итого ≈ 3.7 мин** ✅

Кэш моделей (volumes ollama_models, hf_cache) сохранялся — первая закачка
моделей (~8 ГБ) добавляет время по скорости сети и в замер сознательно
не включена (README предупреждает).
