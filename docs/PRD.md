# PRD — LYRA. Гармония знаний

RAG-платформа корпоративных знаний: единая точка поиска и вопросно-ответного доступа к знаниям компании с обязательными цитатами источников.

**Версия:** 0.1 (MVP) · **Дата:** 2026-07-08 · **Статус:** утверждено для MVP

---

## 0. Assumptions (допущения)

Зафиксированы до начала разработки; изменение любого из них требует ревизии PRD и связанных ADR.

| # | Допущение | Влияние |
|---|-----------|---------|
| A-1 | Язык корпуса и вопросов — преимущественно русский; английский поддерживается | Выбор embeddings/reranker — только мультиязычные модели ([ADR-003](adr/ADR-003-embedding-model.md), [ADR-004](adr/ADR-004-reranker.md)) |
| A-2 | LLM генерации — локальная модель через Ollama, без облачных API в runtime | NFR по latency честно рассчитаны на локальный инференс; провайдер-агностичная абстракция ([ADR-009](adr/ADR-009-llm-provider-abstraction.md)) |
| A-3 | Embeddings и reranker — self-hosted OSS | Нулевая стоимость API, требование к ресурсам ноутбука/сервера |
| A-4 | Источники MVP: загрузка файлов (PDF, DOCX, MD, TXT) + один живой коннектор Confluence | Notion, Google Drive, сетевые диски — out-of-scope MVP |
| A-5 | Демо запускается локально через docker-compose | K8s — только манифесты как артефакт production-трека, без реального кластера |
| A-6 | Аутентификация MVP: JWT + seed-пользователи (создаются миграцией) | SSO/OAuth/SCIM — production-трек |
| A-7 | Один tenant в MVP; `tenant_id` присутствует в схеме БД с первого дня | Мультитенантность в runtime — production-трек |
| A-8 | Демо-корпус: ~100–500 документов, до ~50k chunks | pgvector достаточен; критерии миграции на Qdrant — [ADR-001](adr/ADR-001-vector-store-pgvector-vs-qdrant.md) |
| A-9 | LLM-judge для offline-evals может быть сильнее генератора (допустим облачный judge в CI, но не в runtime) | Качество eval-метрик не ограничено локальной моделью |
| A-10 | Железо: ноутбук разработчика (CPU; GPU опционально ускоряет) | Реалистичные p50/p95 в [nfr.md](nfr.md) |

## 1. Проблема

Знания компании разбросаны по Confluence, Notion, сетевым дискам, PDF-архивам и почте:

- **Поиск не находит.** Штатный поиск (Confluence, файловые шары) работает по ключевым словам, не понимает перефразировок и синонимов, не ищет сквозь несколько систем сразу.
- **Ответам нельзя доверять.** Даже если сотрудник получил ответ (от коллеги, из чат-бота без источников), непонятно, актуален ли он и на чём основан.
- **Время утекает.** Сотрудник тратит десятки минут на поиск того, что уже описано; эксперты отвечают на одни и те же вопросы; онбординг растягивается.

**Гипотеза продукта:** единый вопросно-ответный интерфейс поверх всех источников, который отвечает на естественном языке **и всегда показывает, из какого документа взят ответ**, снижает время поиска и повышает доверие к знаниям.

## 2. Целевая аудитория

Компании от 50 человек (горизонтальный спрос — не привязан к отрасли). Внутри компании:

### Персоны и роли

| Персона | Роль в системе | Потребность |
|---------|----------------|-------------|
| **Сотрудник** (разработчик, поддержка, продажи, HR) | `viewer` | Быстро получить проверяемый ответ на вопрос по внутренним знаниям |
| **Контент-администратор** (владелец базы знаний, техписатель) | `editor` | Подключать источники, загружать документы, следить за полнотой и свежестью индекса, разбирать негативный фидбек |
| **Платформенный администратор** (DevOps/IT) | `admin` | Управлять пользователями и ролями, наблюдать за здоровьем системы, запускать evals, контролировать стоимость |

## 3. Use cases

| ID | Кейс | Персона | Сценарий |
|----|------|---------|----------|
| UC-1 | Вопрос-ответ с цитатами | Сотрудник | Задаёт вопрос в чате → получает ответ со ссылками на документы-источники; кликает по цитате → открывает документ |
| UC-2 | Честный отказ | Сотрудник | Спрашивает то, чего нет в базе → система отвечает «в базе знаний не найдено», а не галлюцинирует |
| UC-3 | Уточняющий диалог | Сотрудник | Задаёт follow-up вопрос в той же сессии; система учитывает историю диалога |
| UC-4 | Загрузка документов | Контент-админ | Загружает PDF/DOCX/MD через UI или API → документы асинхронно парсятся и индексируются, виден статус job |
| UC-5 | Подключение Confluence | Контент-админ | Настраивает коннектор (URL, токен, spaces) → первичная и периодическая инкрементальная синхронизация |
| UC-6 | Обновление документа | Контент-админ | Заливает новую версию → старые chunks заменяются, повторная загрузка того же файла не создаёт дубликатов (идемпотентность) |
| UC-7 | Фидбек | Сотрудник | Ставит 👍/👎 ответу, опционально комментирует → фидбек виден админу и попадает в eval-контур |
| UC-8 | Запуск evals | Платф. админ | Запускает offline-eval на golden dataset → видит faithfulness / relevance / precision / recall и сравнение с baseline |
| UC-9 | Прямой поиск | Сотрудник / интеграции | Выполняет поисковый запрос без генерации (raw retrieval) — быстрый список релевантных фрагментов |
| UC-10 | Наблюдение | Платф. админ | Смотрит дашборды: latency, объём индекса, стоимость, доля ответов с отказом |

## 4. Scope MVP

### Входит (In-scope)

**Ingest:**
- Загрузка файлов: PDF, DOCX, Markdown, TXT (API + UI).
- Коннектор Confluence: первичная и инкрементальная синхронизация выбранных spaces по расписанию.
- Асинхронный пайплайн (Celery): parse → normalize → chunk → embed → index. Статус job наблюдаем через API/UI.
- Идемпотентность (content_hash) и версионирование документов.
- Базовая PII-фильтрация при ingest (детекция секретов/ключей, отказ в индексации — см. [security-and-access.md](security-and-access.md)).

**Retrieval и генерация:**
- Гибридный поиск: BM25 (PostgreSQL full-text) + векторный (pgvector), слияние через RRF ([ADR-005](adr/ADR-005-hybrid-search-fusion.md)).
- Reranker (bge-reranker-v2-m3, self-hosted).
- LangGraph-пайплайн с узлами: retrieve → grade sufficiency → corrective retrieve (query rewrite, ≤2 итерации) → generate → cite → self-check ([ADR-006](adr/ADR-006-langgraph-topology.md)).
- Обязательные цитаты: каждый фактологический ответ содержит ссылки на chunks-источники; ответ без опоры на контекст запрещён ([ADR-007](adr/ADR-007-citation-strategy.md)).
- Confidence в ответе (из sufficiency-оценки и self-check).
- Стриминг ответа (SSE).

**Приложение:**
- FastAPI REST API ([api-contract.md](api-contract.md)).
- React-чат: диалог, отображение цитат с переходом к источнику, фидбек, страница загрузки документов, статус ingest-jobs.
- JWT-аутентификация, роли viewer/editor/admin (RBAC на уровне эндпоинтов).

**Качество и эксплуатация:**
- Offline evals: faithfulness, answer relevance, context precision/recall; golden dataset; порог в CI ([eval-plan.md](eval-plan.md)).
- Наблюдаемость: структурные логи, Prometheus-метрики, Grafana-дашборд, трейсинг LLM-вызовов.
- docker-compose для полного стека; GitHub Actions CI (lint, tests, evals-gate).

### Не входит (Out-of-scope MVP) — production-трек

| Отложено | Куда | Причина |
|----------|------|---------|
| SSO/OAuth/SCIM | Production фаза P1 | Для демо достаточно JWT |
| RBAC на уровне документов (ACL при retrieval) | Production фаза P1 | Схема заложена, enforcement выключен |
| Мультитенантность в runtime | Production фаза P2 | `tenant_id` в схеме есть, изоляция не включена |
| Коннекторы Notion, Google Drive, сетевые диски | Production фаза P2 | Интерфейс `SourceConnector` рассчитан на расширение |
| Миграция на Qdrant | Production фаза P3, по критериям [ADR-001](adr/ADR-001-vector-store-pgvector-vs-qdrant.md) | pgvector достаточен для демо-масштаба |
| Kubernetes + автоскейл | Production фаза P3 | Манифесты пишутся, кластер не разворачивается |
| Граф-RAG, агентный мульти-source поиск | Production фаза P4 | Исследовательские улучшения качества |
| Онлайн-дообучение / fine-tuning моделей | Не планируется в горизонте | Вне ценности кейса |
| Мобильное приложение | Не планируется | Web-чат достаточен |

## 5. Функциональные требования

### Ingest
- **FR-1.** Система принимает загрузку файлов PDF/DOCX/MD/TXT размером до 50 МБ через API и UI.
- **FR-2.** Ingest выполняется асинхронно; API немедленно возвращает `job_id`; статус job доступен по API (`queued/processing/completed/failed` + причина ошибки).
- **FR-3.** Повторная загрузка идентичного содержимого (тот же `content_hash`) не создаёт новых chunks и завершается со статусом `skipped_duplicate`.
- **FR-4.** Обновлённый документ создаёт новую версию; chunks предыдущей версии атомарно исключаются из выдачи.
- **FR-5.** Коннектор Confluence синхронизирует выбранные spaces: первичная полная загрузка + инкрементальная по расписанию (изменённые/удалённые страницы).
- **FR-6.** При ingest выполняется детекция секретов (API-ключи, пароли, приватные ключи); документ с секретами не индексируется, админ получает уведомление в журнале.

### Retrieval и ответы
- **FR-7.** Запрос пользователя обрабатывается гибридным поиском (BM25 + вектор, RRF) с последующим reranking.
- **FR-8.** Перед генерацией система оценивает достаточность найденного контекста; при недостаточности — переформулирует запрос и повторяет поиск (до 2 итераций).
- **FR-9.** Если контекст недостаточен после corrective-итераций, система возвращает честный отказ с формулировкой «в базе знаний не найдено» и списком ближайших найденных документов.
- **FR-10.** Каждый ответ содержит: текст с inline-маркерами цитат `[n]`, массив citations (документ, ссылка, цитируемый фрагмент), confidence.
- **FR-11.** Ответ проходит self-check на faithfulness; при провале — ответ регенерируется или деградирует до честного отказа.
- **FR-12.** Ответ стримится по SSE; citations и confidence передаются финальным событием.
- **FR-13.** Диалоговая история сессии учитывается при обработке follow-up вопросов (конденсация вопроса).

### Пользователи и доступ
- **FR-14.** Аутентификация по JWT; роли viewer/editor/admin; матрица доступа к эндпоинтам — [security-and-access.md](security-and-access.md).
- **FR-15.** Все запросы и ответы журналируются в аудит-лог (кто, когда, что спросил, что отвечено, какие источники).

### Фидбек и качество
- **FR-16.** Пользователь может оценить ответ (👍/👎) и оставить комментарий; фидбек привязан к сообщению и trace.
- **FR-17.** Админ может запустить offline-eval на golden dataset и увидеть метрики в сравнении с baseline.
- **FR-18.** CI блокирует merge при падении eval-метрик ниже порогов ([eval-plan.md](eval-plan.md)).

### Наблюдаемость
- **FR-19.** Каждый LLM-вызов трейсится: prompt, модель, токены, длительность, узел графа; трейс связан с `trace_id` ответа.
- **FR-20.** Prometheus-метрики: latency по этапам пайплайна, размер индекса, RPS, доля отказов, ошибки ingest.

## 6. Метрики успеха продукта

Для MVP (портфолио-кейс) метрики делятся на демонстрируемые в eval-контуре и целевые продуктовые (для production).

### Качество (измеряются evals, гейт в CI)

| Метрика | Порог MVP | Комментарий |
|---------|-----------|-------------|
| Faithfulness | ≥ 0.85 | Доля утверждений ответа, подтверждённых контекстом |
| Answer relevance | ≥ 0.80 | Соответствие ответа вопросу |
| Context precision | ≥ 0.70 | Доля релевантных chunks в переданном контексте |
| Context recall | ≥ 0.75 | Полнота: найден ли нужный контекст |
| Citation validity | ≥ 0.95 | Доля цитат, реально указывающих на подтверждающий фрагмент |
| Honest refusal rate | ≥ 0.90 | Доля корректных отказов на вопросах вне корпуса (negative set) |

### Продуктовые (production-трек, собираются с MVP)

| Метрика | Цель | Как измеряется |
|---------|------|----------------|
| Answer acceptance rate | ≥ 70% 👍 среди оценённых | Фидбек в UI |
| Доля ответов с цитатами | 100% фактологических ответов | Автоматически из логов |
| Time-to-answer (p95) | См. [nfr.md](nfr.md) | Метрики latency |
| Deflection экспертов | Снижение повторных вопросов экспертам | Опрос/интервью (пост-MVP) |
| Свежесть индекса | Лаг синхронизации Confluence ≤ 1 ч | Метрика коннектора |
