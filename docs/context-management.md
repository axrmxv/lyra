# Управление контекстом — LYRA

Правила формирования контекста: параметры chunking по типам источников, бюджет токенов запроса, дедупликация, длинные документы, таблицы и код. Стратегия chunking обоснована в [ADR-002](adr/ADR-002-chunking-strategy.md); значения ниже — рабочий конфиг (хранится в `collections.chunking_config`, [data-model.md](data-model.md)).

Все размеры — в токенах **токенайзера bge-m3** для chunks и **токенайзера LLM** для бюджета запроса. Русский текст токенизируется плотнее английского (~1.3–1.7 токена/слово) — считать токенами, не символами.

---

## 1. Параметры chunking по типам источников

| Тип источника | Разбиение | Целевой размер | Max | Overlap | Особенности |
|---------------|-----------|----------------|-----|---------|-------------|
| Confluence (HTML→IR) | По заголовкам H1–H4, затем по абзацам | 512 | 768 | 64 (только внутри длинной секции) | Макросы разворачиваются в текст/таблицы; вложения не индексируются в MVP |
| Markdown | По заголовкам, затем по абзацам | 512 | 768 | 64 | Front-matter → метаданные |
| PDF со структурой | По распознанным заголовкам/блокам | 512 | 768 | 64 | Колонтитулы и номера страниц вырезаются; номер страницы → metadata |
| PDF без структуры (fallback) | Рекурсивно: абзац → предложение | 384 | 512 | 64 | Флаг `low_structure` в metadata — кандидат на ручную проверку |
| DOCX | По стилям заголовков | 512 | 768 | 64 | |
| TXT | Рекурсивно по абзацам | 384 | 512 | 64 | |
| Таблица (любой источник) | Атомарно | — | 768; больше → split по строкам с повторением шапки | 0 | Markdown-сериализация; заголовок секции — префиксом |
| Код-блок | Атомарно | — | 768; больше → по функциям/логическим блокам | 0 | Язык → metadata.block_type=code; окружающее описание — префиксом |

**Контекстный заголовок:** каждому chunk перед эмбеддингом добавляется префикс `«{doc_title} > {headings_path}»`. Префикс входит в embedding-текст и в текст для LLM, но не учитывается в overlap.

## 2. Бюджет контекста запроса

Модель: Qwen2.5-instruct, окно 32k ([ADR-009](adr/ADR-009-llm-provider-abstraction.md)). Рабочий лимит — 16k на промпт генерации (латентность CPU растёт с длиной промпта; 32k — запас, не цель).

| Компонент | Бюджет (токены) | Примечание |
|-----------|-----------------|------------|
| System-промпт (правила цитирования, формат) | ≤ 1 000 | фиксирован, версионируется |
| История диалога | ≤ 2 000 | последние N сообщений; при переполнении — только condensed_question |
| Контекст (chunks) | ≤ 8 000 | top-8 × ~768 max + заголовки-префиксы |
| Вопрос (condensed) | ≤ 500 | |
| Резерв на ответ (completion) | ≤ 2 000 | max_tokens генерации |
| **Итого worst case** | **~13 500** | запас до 16k |

`chunks.token_count` посчитан токенайзером bge-m3 при ingest и используется при сборке бюджета как приближение LLM-токенов; расхождение токенайзеров (~±10%) покрывается запасом до 16k — пересчитывать токенайзером LLM на каждый запрос не нужно.

Правила усечения (применяются до вызова LLM, строго в этом порядке):
1. Chunks сортированы по rerank-score; добавляются, пока влезают целиком; chunk никогда не режется при сборке контекста.
2. Не влезло минимум 3 chunks → сокращается история диалога (до condensed_question).
3. Всё равно нет 3 chunks (аномально длинные таблицы) → берём сколько влезает, флаг `context_truncated` в трейс.

Бюджеты вспомогательных вызовов: grade_sufficiency — вопрос + сжатые сниппеты chunks (первые ~120 токенов каждого), ≤ 3k; query rewrite — ≤ 1k; self_check — ответ + только процитированные chunks, ≤ 6k.

## 3. Дедупликация

Два уровня:

1. **Ingest-время:** идентичные документы отсекаются по `content_hash` (идемпотентность, [data-model.md](data-model.md)). Внутри корпуса near-duplicate документы (копии политик в разных spaces) в MVP не дедуплицируются — отмечено как production-улучшение (minhash по chunks).
2. **Query-время (сборка контекста):** после reranker, перед отбором top-8:
   - точные дубли текста chunk (разные версии/копии) — оставляется один с максимальным score;
   - **MMR-отбор** (maximal marginal relevance, λ=0.7) по эмбеддингам кандидатов: штраф за похожесть на уже отобранные chunks — контекст покрывает разные аспекты вопроса, а не 8 вариантов одного абзаца;
   - соседние chunks одного документа (ordinal подряд) склеиваются в один блок контекста (без повторения префикса) — LLM видит связный фрагмент.

## 4. Длинные документы

- Chunk несёт `headings_path` + `doc_title` — модель понимает место фрагмента в документе.
- **Parent-block расширение:** если в top-8 попали ≥2 chunks одной секции, вместо них подставляется родительская секция целиком (если ≤ 1 500 токенов) — меньше рваных границ. Реализация проста: chunks хранят ordinal и headings_path.
- Полный «parent-document retrieval» (маленькие chunks для поиска, крупные для контекста) — production-эксперимент, интерфейс retrieval это допускает.
- Суммаризация документов при ingest в MVP не делается (дорого на CPU, сомнительная добавка) — задел: поле для doc-summary в metadata зарезервировано.

## 5. Таблицы

- Таблица — атомарный chunk, markdown-сериализация, префикс: заголовок секции + подпись таблицы (если есть).
- Широкие таблицы (>~20 колонок) — сериализация «строка = список пар колонка: значение» (устойчивее для LLM, чем разъехавшийся markdown).
- Таблица > max-размера — разбиение по строкам, шапка повторяется в каждом фрагменте, metadata несёт `table_part: i/n`.
- В grade_sufficiency сниппет таблицы — шапка + первые 3 строки.

## 6. Код

- Код-блоки — атомарные chunks (block_type=code), не смешиваются с прозой; язык — в metadata.
- Крупные файлы кода (если появятся как источник) — разбиение по функциям/классам; в MVP код встречается как блоки внутри документации — этого правила достаточно.
- BM25-канал важен для кода (точные имена функций/конфигов) — tsvector строится и для code-chunks; embedding тоже считается (bge-m3 приемлемо кодирует код).

## 7. Кэширование (Redis)

| Что | Ключ | TTL | Инвалидация |
|-----|------|-----|-------------|
| Embedding запроса | sha256(normalized query) | 24 ч | — (детерминирован) |
| Retrieval-результат (после rerank) | sha256(query + collection + filters) | 15 мин | по TTL; при активном ingest допускается краткая несвежесть |
| Ответ целиком | не кэшируется в MVP | — | персонализация/ACL сделают это опасным; решение пересмотреть в production с учётом ACL в ключе |

## 8. Изменение параметров

Любое изменение этого документа (размеры, overlap, top-k, бюджеты) = изменение качества системы: обязательный прогон offline-evals ([eval-plan.md](eval-plan.md)) и, при изменении chunking, переиндексация коллекции (`POST /admin/reindex`). Параметры версионируются в `collections.chunking_config` и в `eval_runs.config_snapshot` — каждый eval-run знает, на каких настройках получен.
