# Сценарий демо LYRA (10 минут)

Перед показом: стек поднят (`make up && make seed-demo`), прогрет Ollama
(любой короткий вопрос заранее), `python scripts/demo_smoke.py` — все PASS.
Открыты вкладки: UI (5173), Grafana (3000), терминал с логами api.

> На CPU-стенде ответ генерируется минуты — первый «прогревочный» вопрос
> задайте до начала показа, а в паузах рассказывайте про архитектуру.

## 1. Ответ с проверяемыми цитатами (2 мин)

**Реплика:** «Спросим то, что есть только во внутренних документах».
**Вопрос:** `Какие тарифы подключения есть для бизнеса?`
**Ожидаемо:** стриминг с индикатором стадий («ищу источники → проверяю
достаточность → отвечаю»), ответ с таблицей тарифов и сносками `[n]`;
клик по сноске → popover с заголовком документа, цитатой и релевантностью.
Бейдж уверенности у ответа.

## 2. Честный отказ (1.5 мин)

**Реплика:** «А теперь вопрос, которого в базе нет — обычный чат-бот тут
галлюцинирует».
**Вопрос:** `Какой размер годового бонуса?`
**Ожидаемо:** отдельное состояние отказа «Ответ не найден в базе знаний»
+ список ближайших документов. Подчеркнуть: это ветка графа (self_check /
grade_sufficiency), а не строка в промпте.

## 3. Кривая формулировка → corrective (1.5 мин)

**Вопрос:** `не пускает в сетку когда не из офиса что делать` (нарочно
разговорно).
**Ожидаемо:** в статусах мелькает «уточняю поиск» (corrective_retrieve).
В терминале показать трейс:

```bash
docker logs lyra-api-1 2>&1 | grep '"llm_call"' | tail -8 | jq -r '[.node,.duration_ms] | @tsv'
```

Виден rewrite запроса и повторный retrieve — система чинит слабый вопрос сама.

## 4. Загрузка документа → вопрос по нему (2 мин)

**Действие:** страница «Документы» → drag-n-drop заранее подготовленного
`.md` с уникальным фактом (например, «кодовое слово проекта — Меркурий-7»).
**Ожидаемо:** job в таблице (в очереди → обработка → готово, секунды),
затем вопрос в чате по этому факту → ответ с цитатой на новый документ.
Подчеркнуть: ingest асинхронный, повторная загрузка того же файла даст
`skipped_duplicate` (идемпотентность).

## 5. Качество измеримо: evals и красный CI (1.5 мин)

**Действие:** показать `evals/reports/eval-report.md` последнего прогона
(таблица метрик vs пороги vs baseline, worst-5 с trace_id).
**Реплика:** «Если кто-то ослабит промпт — например, уберёт „отвечай только
по источникам“ — faithfulness и citation validity падают, и CI блокирует
merge». Показать PR-комментарий eval-гейта или локальный красный
`make eval` (заготовить заранее — прогон долгий).

## 6. Наблюдаемость (1.5 мин)

**Действие:** Grafana → «LYRA — обзор»: разбивка последнего запроса по
узлам графа (retrieve мс, grade, generate — где время живёт), доля отказов,
токены, размер индекса.
**Финальная реплика:** «Всё это — на ноутбуке, без внешних API; на GPU
те же ответы — за секунды, код не меняется».

## Аварийные ответы

- Ollama упал → `docker compose restart ollama`, вопрос из шага 1 повторить
  (KV-кэш прогреется быстро).
- Что-то сломалось до показа → `python scripts/demo_smoke.py` покажет, что именно.
