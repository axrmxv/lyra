# ADR-001: Векторное хранилище — pgvector в MVP, Qdrant как target

**Статус:** принято · **Дата:** 2026-07-08

## Контекст

Нужно хранилище векторов для семантического поиска по chunks (демо-масштаб: ≤ 50k chunks, см. допущение A-8 в [PRD](../PRD.md)). Требования: гибридный поиск (вектор + BM25), фильтрация по метаданным (source, collection, версия документа, задел ACL/tenant), транзакционная согласованность с реляционными данными (версионирование документов), простота эксплуатации в docker-compose. Стек фиксирует PostgreSQL как основную СУБД.

## Варианты

| Критерий | pgvector (в основной PG) | Qdrant (отдельный сервис) |
|----------|--------------------------|---------------------------|
| Согласованность с метаданными | ✅ ACID: chunks, версии, tsvector и вектор в одной транзакции | ❌ Двухфазная запись PG↔Qdrant, нужна синхронизация и reconciliation |
| Гибридный поиск | ✅ BM25 (tsvector) и вектор в одном SQL-запросе / одной БД | ⚠️ Sparse-вектора поддерживаются, но BM25-подход другой; либо BM25 остаётся в PG — два хранилища на запрос |
| Производительность ANN на больших объёмах | ⚠️ HNSW в pgvector хорош до единиц миллионов векторов; деградирует при высоком QPS + фильтрах | ✅ Оптимизированный HNSW, payload-индексы для filtered search, квантизация (scalar/PQ/binary) |
| Фильтрация при поиске | ✅ Обычный SQL WHERE (включая будущий RLS для tenant/ACL) | ✅ Payload-фильтры, но дублирование метаданных из PG |
| Эксплуатация MVP | ✅ Ноль дополнительных сервисов | ❌ +1 сервис, +1 бэкап-контур, +1 точка отказа |
| Масштабирование | ⚠️ Вертикальное; шардирование векторов в PG — боль | ✅ Горизонтальное (шардирование, реплики) |
| Демонстрационная ценность | Показывает прагматизм | Показывает работу со специализированной vector DB |
| Когда применять | Малые/средние корпуса, сильная связность с реляционными данными | Миллионы векторов, высокий QPS, multi-tenant изоляция коллекциями |

## Решение

**MVP: pgvector** (HNSW-индекс, cosine distance) в основной PostgreSQL. **Target: миграция на Qdrant** при достижении критериев.

Обязательная изоляция: весь доступ к векторному поиску — через интерфейс `VectorStore` (методы `upsert_chunks`, `search(query_vector, filters, top_k)`, `delete_by_document`), реализация `PgVectorStore` в MVP, `QdrantVectorStore` в production-треке. Retrieval-код не знает о конкретном хранилище.

Критерии переключения на Qdrant (любые два из):
- объём > 5–10 млн chunks или прогноз роста к этому за 2 квартала;
- p95 векторного поиска > 200 мс при целевом QPS несмотря на тюнинг HNSW;
- filtered search (tenant/ACL) деградирует HNSW-индекс (низкий recall или полный скан);
- потребность в квантизации для экономии памяти.

## Последствия

- (+) Один сервис данных в MVP; транзакционное версионирование документов «бесплатно»; RLS-задел для мультитенантности.
- (+) Миграция сводится к реализации `QdrantVectorStore` + backfill-скрипту (перечитать chunks из PG, залить в Qdrant) — путь описан в [PLAN.md](../../PLAN.md), фаза P3.
- (−) На больших объёмах придётся мигрировать; стоимость заложена и ограничена интерфейсом.
- (−) Нагрузка векторного поиска конкурирует с OLTP в одной СУБД — мониторим отдельными метриками с первого дня.
