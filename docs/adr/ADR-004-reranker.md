# ADR-004: Reranker

**Статус:** принято · **Дата:** 2026-07-08

## Контекст

Гибридный поиск + RRF ([ADR-005](ADR-005-hybrid-search-fusion.md)) возвращает top-50 кандидатов с «шумным» порядком: bi-encoder и BM25 оценивают запрос и документ независимо. В контекст LLM попадает лишь ~8 chunks ([context-management.md](../context-management.md)) — ошибка ранжирования на этом срезе напрямую бьёт по context precision и faithfulness. Cross-encoder читает пару (запрос, chunk) целиком и ранжирует точнее. Требования: русский язык, self-hosted, приемлемая латентность на CPU для 50 пар.

## Варианты

| Критерий | bge-reranker-v2-m3 (self-hosted) | Cohere Rerank 3.5 (API) | Без reranker (только RRF) |
|----------|----------------------------------|--------------------------|---------------------------|
| Русский язык | ✅ Мультиязычный (та же база m3) | ✅ | — |
| Стоимость | ✅ $0, свои ресурсы | ❌ Платный API, данные наружу — против A-3 | ✅ |
| Качество top-k | ✅ Существенный прирост precision@k над RRF | ✅ Сопоставимо/лучше | ❌ Хуже: bi-encoder упускает тонкую релевантность |
| Латентность | ⚠️ CPU: ~0.5–2 с на 50 пар (батчинг), GPU: <100 мс | ✅ ~100–300 мс сеть | ✅ 0 |
| Эксплуатация | +1 контейнер (TEI поддерживает rerank-модели) | Ключ, лимиты, зависимость | — |
| Когда применять | Self-hosted RAG с требованием качества (наш случай) | Облачный стек | Прототип, жёсткий бюджет латентности |

## Решение

**BAAI/bge-reranker-v2-m3**, сервинг через text-embeddings-inference (rerank-эндпоинт) отдельным контейнером — тот же паттерн, что embeddings ([ADR-003](ADR-003-embedding-model.md)).

- Схема: RRF top-50 → reranker → **top-8** в контекст (числа — конфиг, тюнятся по evals).
- Порог отсечения по rerank-score: кандидаты ниже порога не попадают в контекст, даже если top-8 не набран — score также входит в расчёт confidence ([ADR-007](ADR-007-citation-strategy.md)).
- **Graceful degradation:** сервис недоступен/таймаут (бюджет 3 с) → используется RRF-порядок, в трейс и ответ ставится флаг `degraded` ([architecture.md](../architecture.md), точки отказа).
- Латентность reranker на CPU учтена в NFR ([nfr.md](../nfr.md)); при появлении GPU — тот же контейнер с `--device gpu`.

## Последствия

- (+) Заметный прирост context precision — прямо влияет на faithfulness и метрики [eval-plan.md](../eval-plan.md); демонстрирует полный retrieval-стек в портфолио.
- (+) Отсечение по score даёт sufficiency-узлу ([ADR-006](ADR-006-langgraph-topology.md)) дополнительный сигнал «найдено мало хорошего».
- (−) Ещё один stateful-по-памяти сервис (~2 GB RAM) в docker-compose.

## Фактические замеры CPU (фаза 3) и калибровка

Изначальная оценка «top-50 → rerank за 0.5–2 с на CPU» **не подтвердилась**:
замер на живом стенде (WSL2, CPU) — ~0.3–0.5 с/пару при 400–600 символах текста,
25 пар × полные chunks (512 ток.) = **103 с**. Cross-encoder на 568M параметров
на CPU принципиально дороже bi-encoder'а.

Решение (конфиг, дефолты для CPU-стенда): `rerank_top_n=12` (на rerank идёт
голова RRF-списка, хвост сохраняет RRF-порядок ниже), `rerank_text_max_chars=600`
(скорится начало chunk — префикс c заголовками + начало текста),
`reranker_timeout_s=15`. Итог: ~4–6 с на запрос — вписывается в бюджет chat
([nfr.md](../nfr.md)), но /search c rerank на CPU медленнее исходного NFR.
На GPU вернуть: top_n=50, max_chars=4000, timeout=3 c. Пороги качества
контролирует eval-контур фазы 6 — если top-12 с обрезкой просядет по
context precision, ревизия этого раздела.
