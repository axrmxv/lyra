# ADR-005: Гибридный поиск и слияние результатов (RRF vs взвешивание)

**Статус:** принято · **Дата:** 2026-07-08

## Контекст

Один канал поиска недостаточен: векторный поиск ловит перефразировки и смысл, но упускает точные термины (артикулы, имена конфигов, аббревиатуры, код); BM25 — наоборот. Для корпоративных знаний обязательны оба канала. Нужно решить, как сливать два ранжированных списка с несравнимыми score (BM25 — неограниченный, cosine similarity — [0..1] с другой формой распределения).

Каналы MVP:
- **Лексический:** PostgreSQL full-text (`tsvector` с конфигурацией `russian` + `english`, ts_rank_cd) — BM25-класс ранжирования без внешнего движка.
- **Семантический:** pgvector, cosine, HNSW ([ADR-001](ADR-001-vector-store-pgvector-vs-qdrant.md), [ADR-003](ADR-003-embedding-model.md)).

## Варианты

| Критерий | RRF (Reciprocal Rank Fusion) | Взвешенная сумма нормализованных score | Только один канал |
|----------|------------------------------|----------------------------------------|-------------------|
| Нормализация score | ✅ Не нужна — работает по рангам | ❌ Нужна (min-max/z-score), чувствительна к распределению и выбросам | — |
| Гиперпараметры | ✅ Один (k), общепринятый дефолт k=60, малочувствителен | ❌ Веса каналов требуют тюнинга на eval-датасете | — |
| Качество | ✅ Устойчивый сильный бэйзлайн (стандарт индустрии) | ✅ Потенциально выше **после** тюнинга под корпус | ❌ |
| Прозрачность/отладка | ✅ Легко объяснить ранги | ⚠️ Скоры смешаны, сложнее дебажить | ✅ |
| Расширяемость (3+ каналов) | ✅ Тривиально добавить канал (например, sparse bge-m3) | ⚠️ Каждый канал — новый вес | ❌ |
| Когда применять | Нет размеченных данных для тюнинга; старт проекта | Есть eval-датасет и время на тюнинг | Прототипы |

## Решение

**RRF с k=60.** Формула: `score(d) = Σ_каналов 1 / (k + rank_канал(d))`; документ отсутствует в списке канала — канал даёт 0.

- Каждый канал возвращает top-50; после fusion — top-50 → reranker ([ADR-004](ADR-004-reranker.md)) → top-8.
- Оба канала выполняются **параллельно** (asyncio) одним retrieval-слоем; фильтры метаданных (collection, source, задел ACL/tenant) применяются в обоих каналах **до** ранжирования.
- Реализация — на стороне приложения (не SQL-гигант): два запроса, слияние в Python. Проще тестировать и добавлять каналы.
- Взвешенное слияние отмечено как тюнинг production-трека: когда golden dataset ([eval-plan.md](../eval-plan.md)) вырастет, веса подбираются по context recall/precision; интерфейс `Fuser` позволяет заменить стратегию конфигом.

## Последствия

- (+) Ноль тюнинга на старте, воспроизводимое поведение, стандартное решение, которое легко защитить на собеседовании.
- (+) Добавление третьего канала (learned sparse из bge-m3, граф-RAG) не меняет схему слияния.
- (−) RRF игнорирует абсолютную величину score — «уверенность» каналов теряется; компенсируется reranker-score как основным сигналом качества дальше по пайплайну.
- (−) Два поисковых запроса на каждый пользовательский вопрос (и на каждую corrective-итерацию) — учтено в бюджете латентности ([nfr.md](../nfr.md)) и кэше Redis.
