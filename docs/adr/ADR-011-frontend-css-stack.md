# ADR-011: CSS-стек фронтенда для UI-редизайна (Tailwind vs расширение CSS-переменных)

**Статус:** принято · **Дата:** 2026-07-18

## Контекст

Фаза P0 (PLAN.md) — редизайн UI под уровень современных LLM-чатов (гибрид ChatGPT/Claude): единая система дизайн-токенов, светлая и тёмная темы, ре-скин всех страниц, injection-safe markdown. Текущая стилизация — один глобальный `frontend/src/styles.css` (~500 строк) с CSS-переменными в `:root`, без CSS-фреймворков и UI-библиотек; тёмной темы нет.

Ограничения:
- Стек зафиксирован в `.claude/CLAUDE.md` — изменение только через ADR (этот документ).
- Запрет `dangerouslySetInnerHTML` и рендеринг контента как текста (`.claude/rules/frontend.md`) — CSS-решение не должно требовать HTML-строк.
- Prettier + pre-commit — контракт фазы 0; версии плагинов pre-commit пинятся под lockfile (грабли mirrors-prettier).
- Продуктовая честность: рефакторинг стилей не меняет поведение компонентов, Vitest-тесты остаются зелёными.
- Объём CSS вырастет в разы: 2 темы × все страницы × новые компоненты (markdown-типографика, поповеры, кнопки действий). Вопрос — на чём это писать и поддерживать.

## Варианты

| Критерий | Tailwind CSS 4 (utility-first) | Расширение plain CSS + переменные | UI-библиотека (MUI/Chakra) |
|----------|-------------------------------|-----------------------------------|----------------------------|
| Новые зависимости | ⚠️ 2 dev-зависимости (`tailwindcss`, `@tailwindcss/vite`), build-time, 0 runtime | ✅ Ноль | ❌ Крупная runtime-зависимость + theme-engine |
| Дизайн-токены и темизация | ✅ CSS-first `@theme` → те же CSS-переменные; dark-вариант встроен | ✅ Возможно, но каскад тем/состояний пишется и дисциплинируется вручную | ⚠️ Токены в JS-теме, свой DSL |
| Скорость итераций редизайна | ✅ Стили в разметке, нет прыжков файл↔файл, нет мёртвого CSS | ❌ Каждый элемент — именованный класс + правило; мёртвые правила копятся | ✅ Готовые компоненты, но «чужой» вид |
| Консистентность | ✅ Утилити принуждают к шкале токенов | ⚠️ Держится на ревью | ✅ |
| Вид «как ChatGPT/Claude» | ✅ Полный контроль | ✅ Полный контроль | ❌ Выглядит как Material/Chakra |
| Риск для инвариантов (тесты, a11y, layering) | ✅ Только классы в разметке | ✅ Только CSS | ❌ Подмена компонентов = смена поведения и тестов |
| Издержки инструментария | ⚠️ Vite-плагин, prettier-plugin-tailwindcss (пин в pre-commit) | ✅ Нет | ❌ Настройка theme provider, SSR-нюансы |
| Когда применять | Много нового UI, нужна скорость и консистентность без UI-kit | Точечные правки существующего UI, минимализм зависимостей | Админки/внутренние тулзы, где вид не важен |

## Решение

**Tailwind CSS 4 с CSS-first конфигурацией.** Объём нового UI (2 темы, все страницы, markdown-типографика) делает ручной CSS-каскад дороже, чем дисциплина утилити-классов; UI-библиотека отвергнута как несущая чужой визуальный язык и риск для тестов.

Реализация:
- `tailwindcss` + `@tailwindcss/vite` (dev-зависимости); `styles.css` остаётся единой точкой входа: `@import "tailwindcss"` + токены + component-layer.
- **Токены — в `@theme`** (цвет, типографика, радиусы, тени, отступы): компилируются в те же CSS-переменные, преемственно текущему `:root`-подходу. Палитра: тёплая гамма читаемости (Claude) на структуре ChatGPT.
- **Тёмная тема:** дефолт от `prefers-color-scheme`, явный переключатель ставит `data-theme` на `<html>`; dark-вариант Tailwind настраивается на `[data-theme="dark"]` через `@custom-variant`.
- **Повторяющиеся паттерны** (bubble, citation-chip, popover, card) — семантические классы в `@layer components` (обычный CSS с `var()`-токенами), а не длинные цепочки `@apply`; одноразовые стили — утилити в разметке.
- **Prettier:** `prettier-plugin-tailwindcss` для сортировки классов; версия пинится в `additional_dependencies` pre-commit-хука (иначе цикл переформатирований — известные грабли).
- **Миграция:** в рамках под-фаз 2–6 P0, страница за страницей; по завершении P0 легаси-правил в `styles.css` не остаётся — двух систем стилей длительно не существует.
- **Границы:** без runtime CSS-in-JS, без UI-kit; классы — только в JSX (`class`-строки не собираются из пользовательского/LLM-контента).

## Последствия

- (+) Скорость и консистентность ре-скина: токены навязываются шкалой утилит, мёртвый CSS не копится, dark mode — штатный механизм.
- (+) Ноль runtime-зависимостей: в бандл попадают только использованные классы; CSP/безопасность не затронуты.
- (+) CSS-переменные сохраняются как контракт темизации — откат к plain CSS возможен без смены токенов.
- (−) Две dev-зависимости и Vite-плагин; смягчение: build-time only, мажорные обновления Tailwind — осознанно, не автобампом.
- (−) Читаемость длинных utility-цепочек в JSX; смягчение: `@layer components` для повторяющихся паттернов, сортировка классов prettier-плагином.
- (−) Порог входа для контрибьюторов, не знающих Tailwind; смягчение: токены и компонентные классы документированы в самом `styles.css`, конвенции — в `.claude/rules/typescript.md` после принятия.
- (−) prettier-plugin-tailwindcss в pre-commit — ещё один пин версии; смягчение: тот же механизм `additional_dependencies`, что уже применён для prettier.
