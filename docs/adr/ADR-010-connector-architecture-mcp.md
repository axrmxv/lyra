# ADR-010: Архитектура коннекторов источников (MCP-подход)

**Статус:** принято · **Дата:** 2026-07-08

## Контекст

MVP включает загрузку файлов и один живой коннектор — Confluence ([PRD](../PRD.md), A-4); target — Notion, Google Drive, диски. Требования: инкрементальная синхронизация, идемпотентность, обработка удалений, единый вход в общий ingest-пайплайн ([architecture.md](../architecture.md) §2). Проект декларирует MCP-коннекторы как часть AI-ядра — нужно определить, где MCP уместен.

## Варианты

| Критерий | Единый интерфейс `SourceConnector` + MCP-обёртка (выбран) | Прямые интеграции ad-hoc в задачах Celery | Только MCP-серверы для всех источников |
|----------|-----------------------------------------------------------|-------------------------------------------|----------------------------------------|
| Расширяемость на новые источники | ✅ Новый класс по контракту | ❌ Копипаста логики синхронизации | ✅ |
| Инкрементальность/идемпотентность | ✅ В базовом классе один раз | ❌ Реализуется каждый раз заново | ⚠️ MCP-протокол не решает сам |
| Сложность MVP | ✅ Умеренная | ✅ Минимальная, но техдолг сразу | ❌ MCP-сервер на каждый источник — оверхед для одного Confluence |
| Демонстрация MCP в продукте | ✅ Есть, дозированно | ❌ | ✅ Максимальная |

## Решение

**Единый интерфейс `SourceConnector`** (внутренний контракт платформы), Confluence как первая реализация, экспонируемая также как MCP-сервер:

```
class SourceConnector(Protocol):
    async def list_changes(self, cursor: SyncCursor | None) -> ChangeSet
        # ChangeSet: added/updated/deleted external_id + updated_at + новый cursor
    async def fetch(self, external_id: str) -> RawDocument
        # содержимое + метаданные (title, url, author, updated_at, space/папка)
    def normalize(self, raw: RawDocument) -> DocumentIR
        # единое промежуточное представление (ADR-002)
```

- **Инкрементальность:** `SyncCursor` (для Confluence — CQL `lastModified >= X` + пагинация) хранится в `sources.sync_cursor`; beat-задача ([ADR-008](ADR-008-task-queue-celery-vs-rq.md)) вызывает `list_changes` и ставит per-document задачи в очередь `ingest`.
- **Идемпотентность:** ключ документа — `(source_id, external_id)`; содержимое сверяется по `content_hash` до chunking; неизменённый документ — no-op. Повторный полный sync безопасен.
- **Удаления:** `deleted` из ChangeSet помечает документ `status=deleted`, chunks исключаются из выдачи атомарно (тот же механизм версионирования, [data-model.md](../data-model.md)).
- **MCP-роль:** Confluence-коннектор оформляется как MCP-сервер (tools: `list_spaces`, `search_pages`, `get_page`), которым пользуется и ingest-пайплайн, и — в production-треке P4 — агентный поиск по живым источникам. В MVP это тонкая обёртка над теми же методами: демонстрирует MCP без усложнения пайплайна.
- Загрузка файлов — вырожденный коннектор `UploadConnector` (без list_changes; вход через API), но с теми же normalize и идемпотентностью.
- Секреты коннекторов (токен Confluence) — только в env/secret-хранилище, не в БД открытым текстом ([security-and-access.md](../security-and-access.md)).

## Последствия

- (+) Notion/Drive в production-треке — новые реализации контракта без изменения пайплайна; синхронизационная логика написана один раз.
- (+) MCP-сервер Confluence переиспользуется агентным retrieval в P4 — задел без лишней работы сейчас.
- (−) Абстракция чуть дороже прямой интеграции на старте (~ один лишний слой) — цена расширяемости.
- (−) Confluence API rate limits определяют скорость первичной синхронизации — backoff и настройка частоты beat обязательны.
