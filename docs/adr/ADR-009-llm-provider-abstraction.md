# ADR-009: LLM-провайдер и абстракция

**Статус:** принято · **Дата:** 2026-07-08

## Контекст

По решению запуска ([PRD](../PRD.md), допущение A-2) генерация в MVP — локальная модель через Ollama: демо без API-ключей и затрат, полностью офлайн. При этом путь до production почти наверняка означает облачный API или GPU-кластер — код не должен зависеть от провайдера. LLM используется в нескольких ролях ([ADR-006](ADR-006-langgraph-topology.md)): генерация (streaming), grading/self-check (structured output), query rewrite (короткие вызовы).

## Варианты

### Модель для Ollama

| Критерий | Qwen2.5-instruct 7B (выбор по умолчанию) | Qwen2.5-instruct 14B | Llama 3.1 8B |
|----------|------------------------------------------|----------------------|--------------|
| Русский язык | ✅ Один из лучших в классе | ✅ Лучше 7B | ⚠️ Заметно слабее на русском |
| Скорость на CPU | ✅ Приемлема (q4-квантизация) | ❌ Медленно на CPU, нужен GPU | ✅ |
| Следование формату (JSON, маркеры цитат) | ✅ Хорошее | ✅ | ⚠️ |
| Контекст | 32k (достаточно, [context-management.md](../context-management.md)) | 32k | 128k |

### Слой абстракции

| Критерий | Свой тонкий интерфейс `LLMClient` (выбран) | LiteLLM | Прямые вызовы SDK по коду |
|----------|--------------------------------------------|---------|---------------------------|
| Зависимости | ✅ Ноль лишних | ⚠️ Крупная зависимость, широкая поверхность | ✅ |
| Контроль трейсинга/ретраев | ✅ Полный, в одном месте | ⚠️ Через их каллбеки | ❌ Размазано по коду |
| Скорость смены провайдера | ✅ Новая реализация интерфейса | ✅ Смена строки конфига | ❌ Переписывание |
| Ценность для портфолио | ✅ Показывает проектирование границ | ⚠️ | ❌ |

## Решение

1. **Ollama + Qwen2.5-instruct 7B (q4)** как дефолт MVP; 14B — опция конфига при наличии GPU. Модель фиксируется тегом в конфиге, не latest.
2. **Собственный интерфейс `LLMClient`** (protocol): `chat(messages, ...) -> str`, `chat_stream(...) -> AsyncIterator[str]`, `structured(messages, schema: type[BaseModel]) -> BaseModel`. Реализации: `OllamaClient` (MVP), `AnthropicClient`/`OpenAIClient` (production-трек, за тем же интерфейсом). LiteLLM отклонён как лишняя зависимость при трёх нужных методах.
3. **Трейсинг обязателен на уровне интерфейса** (FR-19): декоратор/обёртка пишет prompt, модель, параметры, токены (из ответа Ollama), длительность, имя узла графа, trace_id — в структурный лог и Prometheus. Ни один LLM-вызов не минует обёртку.
4. **Structured output:** Ollama JSON-mode (`format: json_schema`) + валидация Pydantic + 1 retry с сообщением об ошибке валидации — для grading/self-check/rewrite.
5. Разные роли могут использовать разные модели (конфиг: `generation_model`, `grading_model`) — grading можно перевести на меньшую/быструю модель, не трогая код.
6. Таймауты и деградация: генерация — таймаут 60 с, 1 retry; недоступность Ollama → честная 503 ([architecture.md](../architecture.md)).

## Последствия

- (+) Демо запускается одним `docker compose up` без ключей; $0 стоимость запроса ([nfr.md](../nfr.md)).
- (+) Миграция на облако в production — реализация одного класса + конфиг; NFR-таблица target уже считает облачный вариант.
- (−) Качество 7B ниже frontier-моделей: строже промпты, обязательная валидация формата (узел cite), эвристики в grading; это осознанная демонстрация «работы с ограничениями LLM».
- (−) CPU-инференс определяет p95 латентности — честно зафиксировано в NFR; GPU ускоряет без изменения архитектуры.
