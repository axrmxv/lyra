# ADR-002: Стратегия chunking

**Статус:** принято · **Дата:** 2026-07-08

## Контекст

Качество RAG критически зависит от разбиения документов: слишком мелкие chunks теряют контекст, слишком крупные размывают релевантность и раздувают бюджет токенов (важно для локальной LLM с контекстом 32k, [ADR-009](ADR-009-llm-provider-abstraction.md)). Источники разнородны: Confluence (HTML со структурой), Markdown, PDF (слабая структура), DOCX; внутри — таблицы и код-блоки. Корпус преимущественно русскоязычный. Детальные параметры по типам источников — [context-management.md](../context-management.md).

## Варианты

| Критерий | Fixed-size (по токенам) | Recursive structure-aware (по структуре, с fallback на токены) | Semantic (по сходству эмбеддингов соседних предложений) |
|----------|------------------------|----------------------------------------------------------------|--------------------------------------------------------|
| Сохранение смысловых границ | ❌ Режет посреди раздела/предложения | ✅ Заголовки, абзацы, списки — естественные границы | ✅ Лучшие смысловые границы |
| Стоимость ingest | ✅ Дёшево | ✅ Дёшево (парсинг структуры уже есть) | ❌ Эмбеддинг каждого предложения — дорого на CPU |
| Предсказуемость и отладка | ✅ | ✅ Детерминированно | ⚠️ Недетерминированность при смене модели эмбеддингов |
| Таблицы и код | ❌ Разрывает | ✅ Обрабатываются как атомарные блоки | ⚠️ Требует спец-обработки всё равно |
| Когда применять | Логи, неструктурированный сырой текст | Документы со структурой (наш основной случай) | Исследовательская оптимизация при наличии GPU |

## Решение

**Recursive structure-aware chunking** поверх нормализованного промежуточного представления (`DocumentIR` — дерево: заголовки → секции → блоки {параграф, таблица, код, список}):

1. Разбиение в первую очередь по границам заголовков (H1–H4); путь заголовков (`headings_path`) сохраняется в метаданных chunk и **добавляется префиксом в текст chunk перед эмбеддингом** («контекстный заголовок») — дешёвый вариант contextual retrieval.
2. Целевой размер **~512 токенов**, максимум 768; overlap **10–15%** (~64 токена) только при разрезании длинной секции по абзацам/предложениям.
3. **Таблицы** — атомарный chunk (markdown-сериализация целиком + заголовок секции); таблица больше max-размера — разбивается по строкам с повторением шапки.
4. **Код-блоки** — атомарный chunk с префиксом-описанием из окружающего текста; не смешиваются с прозой.
5. Токенизация считается токенайзером embedding-модели (bge-m3), не символами — критично для русского (в среднем больше токенов на слово, чем в английском).
6. Параметры (размер, overlap) — конфиг per source_type, зафиксированы в [context-management.md](../context-management.md); изменение параметров = переиндексация и прогон evals.

Semantic chunking отклонён для MVP из-за стоимости на CPU и недетерминированности; отмечен как эксперимент production-трека при наличии GPU.

## Последствия

- (+) Chunks соответствуют смысловым единицам документации; headings_path даёт и фильтрацию, и «дешёвый контекст» для эмбеддинга.
- (+) Детерминированность: одинаковый вход → одинаковые chunks (важно для идемпотентности и content_hash).
- (−) Требуется качественный structure-парсинг PDF (этап parse в ingest) — самая хрупкая часть; закладываем fallback: PDF без структуры режется рекурсивно по абзацам.
- (−) Изменение стратегии chunking инвалидирует индекс — процедура переиндексации обязана быть в ingest-пайплайне с первого дня (переиндексация как обычный ingest новой «версии»).
