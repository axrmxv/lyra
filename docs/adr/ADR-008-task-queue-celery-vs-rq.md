# ADR-008: Очередь фоновых задач — Celery vs RQ, выбор брокера

**Статус:** принято · **Дата:** 2026-07-08

## Контекст

Ingest обязан быть асинхронным (антипаттерн — тяжёлый парсинг/эмбеддинг в HTTP-запросе): парсинг PDF, эмбеддинг батчей на CPU — секунды-минуты. Дополнительно нужны: периодическая синхронизация Confluence по расписанию ([ADR-010](ADR-010-connector-architecture-mcp.md)), retry с backoff (недоступность embeddings-сервиса, лимиты Confluence API), rate limiting эмбеддинга, запуск eval-прогонов. Redis уже в стеке (кэш).

## Варианты

### Фреймворк

| Критерий | Celery | RQ |
|----------|--------|-----|
| Retry-политики | ✅ Гибкие: autoretry_for, exponential backoff, jitter, max_retries | ⚠️ Базовые (Retry с интервалами), беднее |
| Расписание | ✅ Celery beat встроен — нужен для синхронизации Confluence | ❌ Отдельный пакет rq-scheduler |
| Приоритеты / маршрутизация очередей | ✅ Несколько очередей, routing, rate_limit per task | ⚠️ Несколько очередей есть, rate limit — нет |
| Надёжность доставки | ✅ acks_late — задача не теряется при смерти воркера | ⚠️ Слабее (job в обработке при падении требует ручного requeue) |
| Простота | ❌ Сложнее конфигурация, известные острые углы | ✅ Заметно проще |
| Наблюдаемость | ✅ Flower, события, зрелые Prometheus-экспортеры | ⚠️ rq-dashboard, скромнее |
| Windows-разработка | ⚠️ Официально не поддерживается нативно — но у нас всё в Docker | ❌ Не работает на Windows нативно (fork) — тоже решается Docker |
| Когда применять | Расписания, retry, rate limit, приоритеты (наш случай) | Простые очереди «поставил-выполнил» |

### Брокер (для Celery)

| Критерий | Redis | RabbitMQ |
|----------|-------|----------|
| Уже в стеке | ✅ Да (кэш) — ноль новых сервисов | ❌ +1 сервис в compose |
| Гарантии доставки | ⚠️ Достаточные с acks_late + visibility_timeout; идемпотентность задач обязательна | ✅ Строже (confirm, DLX из коробки) |
| Масштаб MVP (десятки задач/мин) | ✅ С запасом | ✅ Избыточно |
| Когда применять | Малые/средние нагрузки, Redis уже есть | Высокие требования к доставке, сложный routing |

## Решение

**Celery + Redis-брокер.** RQ отклонён: отсутствие встроенного beat и слабые retry-политики означали бы самописную обвязку ровно там, где Celery даёт готовое; RQ честно отмечен как достаточный для MVP без коннекторов. RabbitMQ отклонён для MVP: не добавляет ценности на нашем масштабе, но добавляет сервис; переход возможен сменой `broker_url` без изменения кода задач.

Конфигурация:
- Очереди: `ingest` (parse/chunk/embed/index), `sync` (коннекторы, beat), `evals` (низкий приоритет).
- `acks_late=True` + **идемпотентные задачи** (шаги ingest переживают повтор: upsert по ключам, проверка content_hash) — компенсирует редкие повторные доставки Redis-брокера.
- `rate_limit` на задачу эмбеддинга — защита embeddings-сервиса.
- Retry: exponential backoff + jitter, max_retries=5 для сетевых ошибок; ошибки парсинга — не ретраятся (permanent failure → `ingest_jobs.status=failed`).
- Beat: расписание синхронизации Confluence (конфиг per source).
- Метрики воркеров в Prometheus (длина очереди, длительность задач, доля ошибок) — часть FR-20.

## Последствия

- (+) Один инфраструктурный сервис (Redis) обслуживает кэш и брокер; готовые beat/retry/rate-limit.
- (+) Идемпотентность задач, навязанная выбором Redis-брокера, — правильное свойство независимо от брокера.
- (−) Celery-конфигурация сложнее RQ — фиксируется один раз в фазе 2 ([PLAN.md](../../PLAN.md)) и копируется как паттерн.
- (−) Redis становится критичным для ingest (не для чтения/чата) — отражено в точках отказа [architecture.md](../architecture.md).
