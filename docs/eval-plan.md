# План оценки качества (Evals) — LYRA

Evals — центральная часть кейса: без метрик заявления «RAG работает» непроверяемы. Контур: offline-evals на golden dataset, гейт в CI, регресс-контроль, пополнение датасета из фидбека. Целевые пороги согласованы с [PRD §6](PRD.md).

---

## 1. Метрики

| Метрика | Что измеряет | Как считается | Порог CI |
|---------|--------------|---------------|----------|
| **Faithfulness** | Доля утверждений ответа, подтверждённых переданным контекстом (анти-галлюцинации) | LLM-judge: декомпозиция ответа на утверждения → проверка каждого против контекста | ≥ 0.85 |
| **Answer relevance** | Отвечает ли ответ на заданный вопрос | LLM-judge: сравнение вопроса и ответа (+обратная генерация вопросов из ответа, cosine к исходному) | ≥ 0.80 |
| **Context precision** | Доля релевантных chunks среди переданных в контекст | LLM-judge per-chunk против вопроса/ground truth; взвешивание по позиции | ≥ 0.70 |
| **Context recall** | Найден ли необходимый контекст | Сопоставление retrieved chunks с `expected_chunk_ids`/`expected_doc_ids` разметки (детерминированно, без judge) | ≥ 0.75 |
| **Citation validity** | Цитата действительно указывает на подтверждающий фрагмент | Детерминированная проверка маркеров (узел cite) + LLM-judge поддержки утверждения ([ADR-007](adr/ADR-007-citation-strategy.md)) | ≥ 0.95 |
| **Honest refusal rate** | Корректный отказ на вопросах вне корпуса | Доля refusal=true на negative set | ≥ 0.90 |
| **False refusal rate** | Отказ на отвечаемых вопросах (обратная сторона) | Доля refusal=true на answerable set | ≤ 0.10 |

Вспомогательные (не гейт, наблюдаем в динамике): retrieval hit@k до и после rerank (изолирует качество retrieval от генерации), средняя латентность/LLM-вызовы на запрос, доля corrective-итераций, доля self-check retry.

**LLM-judge:** допущение A-9 ([PRD](PRD.md)) — judge может быть сильнее генератора: в CI допускается облачная модель (ключ — секрет CI), локально — Qwen 14B/32B. Judge никогда не используется в runtime. Промпты judge версионируются вместе с кодом; смена judge-модели = пересчёт baseline (значения несравнимы между judge-моделями).

## 2. Golden dataset

Состав (цель MVP — **100 items**, минимум для первого гейта — 50):

| Подмножество | Доля | Назначение |
|--------------|------|------------|
| Answerable — факт из одного chunk | 40% | Базовый happy path |
| Answerable — синтез из нескольких chunks/документов | 20% | Проверка recall и мульти-chunk генерации |
| Answerable — таблицы и код | 10% | Проверка спец-обработки ([context-management.md](context-management.md)) |
| Paraphrase-пары | 15% | Чувствительность к формулировкам: один ground truth, 2–3 формулировки вопроса; метрики не должны расходиться > 0.1 между парафразами |
| Unanswerable (negative set) | 15% | Честный отказ: вопросы вне корпуса, в т.ч. «соседние» по теме (труднее, чем полностью чужие) |

Схема item — [data-model.md](data-model.md) (`eval_items`): question, ground_truth_answer, expected_doc_ids/expected_chunk_ids, kind.

**Сбор ground truth:**
1. Демо-корпус фиксируется (снапшот документов в репозитории evals — корпус и датасет версионируются вместе).
2. Синтетическая генерация кандидатов: LLM генерирует вопросы по chunk/секции (question + ответ + ссылка на источник).
3. **Обязательное ручное ревью** каждого item: вопрос естественный? ответ верен? разметка источников полна? Синтетика без ревью в golden set не попадает.
4. Пополнение из эксплуатации: 👎-фидбек (FR-16) и случаи провала self_check разбираются, полезные превращаются в items.

## 3. Прогон и гейт в CI

- Eval-run — Celery-задача (очередь `evals`, [ADR-008](adr/ADR-008-task-queue-celery-vs-rq.md)) или CLI `python -m lyra.evals run --dataset golden`; результаты — в `eval_runs`/`eval_records` + JSON-артефакт.
- **CI (GitHub Actions):** прогон на PR, затрагивающих промпты, retrieval, chunking, конфиги моделей (path-фильтр); полный прогон — nightly и перед релизом. Стенд CI поднимает docker-compose с фиксированным снапшотом корпуса.
- **Гейт:** метрика ниже порога → job красный, merge заблокирован. Пороги — в одном конфиг-файле (`evals/thresholds.yaml`); изменение порога — только через PR с обоснованием (правило в `.claude/rules/evals.md`).
- Отчёт в PR-комментарий: таблица метрик vs baseline (дельты), список worst items со ссылками на трейсы.

## 4. Регресс-контроль

- **Baseline** — агрегаты последнего успешного run на main; хранится как артефакт + запись в `eval_runs`.
- Каждый run пишет `config_snapshot` (git_ref, версии промптов, модель генерации/judge, retrieval-параметры, chunking_config) — любой результат воспроизводим и атрибутируем.
- Правило регресса: падение любой гейт-метрики > 0.05 относительно baseline — красный, даже если абсолютный порог пройден.
- Датасет растёт только добавлением (items не редактируются задним числом; исправление = деактивация старого + новый item) — сравнимость run'ов сохраняется.

## 5. Учёт ограничений LLM

| Ограничение | Как учитываем |
|-------------|---------------|
| **Галлюцинации** | Faithfulness-метрика + runtime self_check ([ADR-006](adr/ADR-006-langgraph-topology.md)) + citation validity; negative set ловит галлюцинации «из общих знаний» на вопросах вне корпуса |
| **Устаревание знаний модели** | Ответ строится только по контексту (промпт запрещает внутренние знания); корпусные факты специально включают item'ы, противоречащие «общеизвестному» (напр., внутренняя политика ≠ типовая) — ловим подмену контекста памятью модели |
| **Чувствительность к формулировкам** | Paraphrase-подмножество: расхождение метрик между парафразами > 0.1 — сигнал нестабильности промпта/retrieval |
| **Недетерминированность** | temperature=0 для генерации в evals; judge — temperature=0; при остаточной дисперсии — медиана из 3 прогонов для спорных items (nightly) |
| **Слабость 7B-grading** | Метрика качества самого grade_sufficiency: на датасете размечаем «должен ли был сработать corrective» и меряем точность узла отдельно |

## 6. Что в MVP, что дальше

| MVP | Production-трек |
|-----|-----------------|
| Offline golden dataset 50–100 items, гейт в CI, baseline-сравнение | Датасет 500+, стратификация по коллекциям |
| Ручной разбор 👎-фидбека | Полуавтоматический контур: кластеризация фидбека, авто-кандидаты в датасет |
| Метрики качества per-run | Online-мониторинг: доля refusal, confidence-распределение, canary-промпты в проде |
| Judge в CI (облачный допустим) | Калиброванный локальный judge + периодическая сверка с сильной моделью |
